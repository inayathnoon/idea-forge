# PRD Specification

Product Requirements Document — defines the what, who, why, and how of the MVP.

## Required Sections

1. **Problem Statement** (1-2 paragraphs)
   - What pain does this solve?
   - For whom?
   - Current workarounds and limitations

2. **Users & Context** (1-2 paragraphs)
   - Primary user persona(s)
   - Secondary users
   - Use context (where, when, how they use it)

3. **Solution Overview** (1-2 paragraphs)
   - What does the product do?
   - How does it solve the problem?
   - Key differentiator

4. **MVP Features** (5-8 features)
   - Feature name
   - 1-sentence description of what it does
   - Why it's in MVP (solves key problem, enables others, etc.)
   - Acceptance criteria (how to know it's done)

5. **Out of Scope** (1-2 items)
   - What we're explicitly NOT doing in v1
   - Why it's deferred
   - When we might revisit

6. **Success Metrics** (3-5 metrics)
   - How do we know the MVP worked?
   - Measurable, tied to user behavior
   - Not vanity metrics

7. **Constraints** (1-3 constraints)
   - Team size, timeline, budget, technical limitations
   - Why they matter

8. **Tech Stack Summary** (optional, brief)
   - Frontend, backend, database
   - One sentence explaining each choice

## Quality Criteria

- ✅ Written for builders, not stakeholders
- ✅ Every feature has a reason to be in MVP
- ✅ Success metrics are measurable
- ✅ No hand-waving ("make it user-friendly" is not a metric)
- ✅ Constraints are specific, not generic
- ✅ Someone could start building from this alone

## Common Pitfalls

- ❌ Features without acceptance criteria
- ❌ "Nice to have" features sneaking into MVP
- ❌ Metrics that are opinions not measurements
- ❌ Tech stack deciding features
- ❌ Treating PRD as marketing document
